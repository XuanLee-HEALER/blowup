# blowup 产品理念讨论

## 当前三个模块

1. **知识库** — 以影人和流派为主线，作者论视角构建电影世界
2. **电影库** — YTS 公共领域资源 + 离线管理 + 嵌入播放器 + 字幕管理
3. **工具模块** — 字幕编辑/对齐、视频编辑（考虑移除，偏离主线）

## 核心困惑

为自己量身定做，还是面向外部用户？

### 自用路线的优势

- 理念清晰：作者论 + 上世纪作者电影黄金期 + YTS 公共领域
- 不用妥协：知识库结构就是你的电影观，不需要迎合大众
- 技术纯粹：不需要处理版权问题、不需要适配各种资源站

### 外部产品的困境

- 电影库对外意义不大：多数人看新片，新片 = 盗版资源 = 法律问题
- 知识库结构高度个人化：作者论是一种立场，不是所有人的立场
- 用户群极窄：关注上世纪作者电影的技术型影迷

## 2026-04-08 讨论

### 结论一：工具模块收缩

字幕下载/对齐保留（服务于电影库的观影体验）。字幕编辑和视频编辑移除（偏离核心）。

### 结论二：两个模块的定位

- **知识库 = 产品核心**，类比 Obsidian，做电影领域的元知识构建工具
- **电影库 = 保持当前实现**，YTS 公共领域 + 离线管理 + 嵌入播放器 + 字幕

### 知识库的元构建方向

#### 当前实现（硬编码）

实体类型固定为：people、films、genres
关系类型固定为：person_films(role)、film_genres、person_genres、person_relations(relation_type)
组织视角固定为：作者论（导演为核心节点）

#### 核心认知：电影世界只有两个底层实体

**人（people）和电影（films）**。这是不可再分的原子。

genres/流派/运动（新浪潮、新好莱坞、黑色电影...）不是底层实体，而是**后验的理论标签**——是人在理解电影世界后贴上去的助记符。它们是观点，不是事实。

#### 进一步抽象：统一条目模型

人和电影的区别也只是"类型"不同。底层可以统一为**条目（entry）**：

- 每个条目：名称（name）、标签（tags）、笔记（wiki）
- 条目之间的关系：带类型的边（`A --[关系类型]--> B`）
- 条目的类型（type）可扩展：初始只有"影人"和"电影"，用户可自行增加

关系不是树（父子/并列），而是**有类型的图**。同一部电影可以同时关联到导演（导演关系）和演员（出演关系），语义不同。

#### 目标：Obsidian for Film

核心类比：Obsidian 是 markdown + 双向链接。blowup 的知识库是**条目 + 带类型关系 + 标签**。

#### UI 结构

侧边栏"知识库"下只有两个入口：

1. **Wiki** — 纯粹的知识浏览器：
   - 顶部：tag 列表（点选 = 过滤）
   - 搜索栏（名称/内容搜索）
   - 条目列表，按更新时间排序，所有条目一视同仁
   - 不区分 type，不做定制化卡片。影人和电影的区别完全由用户通过 tag 自行组织
2. **知识图谱** — 关系图视图，从指定节点展开，D3 力导向图渲染，边上标注关系类型

列表和图谱是同一份数据的两种视图，不需要统一结构。

#### 存储方案

**保持 SQLite**，不需要图数据库。

个人知识库的规模（几百~几千条目，几千条关系）远不到 Neo4j 的适用场景。三张表足矣：

```sql
entries:     id, name, wiki, created_at
entry_tags:  entry_id, tag
relations:   from_id, to_id, relation_type
```

- 查"一个条目的所有关联"= 一条 SQL join
- 图谱渲染 = 全量加载到前端交给 D3，无性能问题
- 已有 S3 同步链路可直接复用

### 已确认结论

1. 工具模块收缩：字幕下载/对齐保留，字幕编辑和视频编辑移除
2. 电影库保持当前实现不变
3. 知识库重构为统一条目模型（entry + tags + typed relations）
4. 去掉 type 字段，不做定制化展示，影人/电影的区别完全靠 tag
5. genres 从独立实体降级为标签
6. 存储保持 SQLite，三张表（entries / entry_tags / relations）
7. Wiki 页面：tag 过滤 + 搜索 + 按更新时间排序，所有条目一视同仁
8. 知识图谱：D3 力导向图，从指定节点展开

### 已确认：关系类型完全开放

- 不预设任何关系类型，由用户在使用中创建
- 已创建过的关系类型可管理（查看、复用、删除）
- 创建关系时可从已有类型中选择，或输入新类型
- 存储上不需要额外的表，relations 表的 relation_type 字段就是字符串，去重查询即可得到已有类型列表

### 已确认：电影库与知识库保持弱关联

知识库条目如果对应电影库中的资源，可以通过链接跳转。不做更强的绑定。

### 已确认：不做兼容迁移

开发阶段，直接设计新的三表结构（entries / entry_tags / relations），旧的 people/films/genres 等表废弃。
